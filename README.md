# API отчетов МойСклад

Генерирует Excel-отчет о валовом обороте по образцу из этой папки.

MVP по умолчанию работает в безопасном `mock`-режиме. Боевой МойСклад включается явно через `DATA_SOURCE=moysklad` или параметр `source=moysklad`.

## Как запустить

Самый простой запуск:

```bat
run_api.bat
```

Скрипт сам создаст `.venv`, установит зависимости, создаст `.env` из `.env.example`, если его еще нет, и поднимет API на:

```text
http://127.0.0.1:8010
```

Тесты:

```bat
run_tests.bat
```

Ручной запуск, если нужен:

```powershell
python -m venv .venv
.\.venv\Scripts\pip install -r requirements.txt
Copy-Item .env.example .env
.\.venv\Scripts\python -m uvicorn app.main:app --host 127.0.0.1 --port 8010
```

## Как сейчас мокуются данные

`DATA_SOURCE=mock` стоит по умолчанию. В этом режиме сервис вообще не ходит в МойСклад.

Логика такая:

1. Если для магазина и месяца есть пример Excel в папке проекта, мок читает строки дневной таблицы из этого файла.
   Сейчас это примеры за май 2026 для `Планета`, `Июнь`, `Высотная`.
2. Если подходящего примера нет, мок генерирует стабильные псевдопродажи по ключу `магазин + месяц`.
   То есть один и тот же запрос всегда даст одинаковые цифры.
3. Генератор отчета потом упаковывает эти данные в такой же Excel-шаблон.

Это нужно, чтобы до получения боевого ключа тестировать API, Excel-формат, асинхронные задачи и скачивание файлов без риска испортить реальные данные.

## Основные ручки

Проверка сервиса:

```http
GET /health
```

Список локально настроенных магазинов:

```http
GET /stores
```

Сейчас заведены: `Планета`, `Июнь`, `Высотная`. Магазины настраиваются в `stores.json`.

Создать задачу генерации отчета:

```http
POST /reports/gross-turnover
Content-Type: application/json

{
  "store": "Планета",
  "month": "2026-05",
  "source": "mock"
}
```

Ответ:

```json
{
  "job_id": "abc123",
  "status": "queued",
  "store": "Планета",
  "month": "2026-05",
  "source": "mock",
  "status_url": "/reports/jobs/abc123",
  "download_url": "/reports/jobs/abc123/download"
}
```

Проверить статус:

```http
GET /reports/jobs/{job_id}
```

Скачать готовый файл:

```http
GET /reports/jobs/{job_id}/download
```

Синхронная ручка оставлена для ручной проверки:

```http
GET /reports/gross-turnover?store=Планета&month=2026-05&source=mock
```

## Диагностика МойСклад

Когда дадут боевой доступ, сначала заполнить `.env`:

```env
MOYSKLAD_TOKEN=...
# или
MOYSKLAD_LOGIN=...
MOYSKLAD_PASSWORD=...
```

Потом дернуть диагностическую ручку:

```http
GET /moysklad/stores
```

Она делает только `GET`-запрос к МойСклад и возвращает торговые точки:

```json
{
  "stores": [
    {
      "id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
      "name": "Планета",
      "href": "https://api.moysklad.ru/api/remap/1.2/entity/retailstore/...",
      "archived": false,
      "updated": "2026-06-14 10:00:00.000"
    }
  ]
}
```

После этого ID нужных точек надо перенести в `stores.json` в поле `moysklad_retail_store_id`. Так продовый отчет будет фильтровать продажи по точному ID торговой точки, а не угадывать по названию.

## Боевой режим

Для боевой генерации:

```json
{
  "store": "Планета",
  "month": "2026-05",
  "source": "moysklad"
}
```

Клиент МойСклад в проекте использует только `GET`-запросы. Перед первым боевым запуском нужно сверить один месяц с отчетом, собранным вручную.
